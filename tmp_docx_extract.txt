Hệ thống game hóa hành động xanh trong trường đại học

• Sinh viên tham gia các nhiệm vụ xanh như tái chế, check-in trạm xanh, báo cáo điểm rác, tham gia sự kiện dọn rác để nhận điểm, huy hiệu, chứng nhận và thứ hạng.

• Backend dùng Java/Spring Boot thay cho .NET nhưng vẫn giữ các nguyên tắc kiến trúc tương đương project thực hành EShopMicroservices.

1. Căn cứ thiết kế và mục tiêu microservices
Mục tiêu của đề tài không phải là sao chép nghiệp vụ EShop, mà là đạt mức độ microservices tương đương: tách service theo bounded context, mỗi service có database riêng, giao tiếp qua API/event rõ ràng, có API Gateway, gRPC, RabbitMQ, Redis, CQRS, DDD/Clean Architecture, Docker Compose và health check.
Các yêu cầu kiến trúc rút ra từ tài liệu môn học
• Microservices là các service nhỏ, độc lập, loosely coupled, có thể deploy độc lập và giao tiếp qua API rõ ràng.
• Mỗi service nên sở hữu database riêng, không để service khác đọc hoặc ghi trực tiếp vào database của mình.
• Project thực hành EShopMicroservices nhấn mạnh CQRS, DDD, Clean Architecture, Vertical Slice, Redis cache, gRPC, RabbitMQ, API Gateway và Docker Compose.
• Cần tránh distributed monolith: tách nhiều service nhưng vẫn dùng chung database, phụ thuộc chéo domain model, hoặc gọi đồng bộ dây chuyền quá dài.
Tiêu chí
Cách EcoQuest đáp ứng
Service độc lập
Mỗi bounded context là một Spring Boot service/container riêng.
Database per service
Catalog, Action, Policy, Reward, Leaderboard, Recognition có DB riêng.
API Gateway
Spring Cloud Gateway route request từ frontend đến services.
Sync communication
Eco Action gọi Verification Policy bằng gRPC.
Async communication
RabbitMQ truyền event giữa Action, Reward, Leaderboard, Recognition.
Cache
Redis dùng cho draft/session/idempotency và leaderboard ranking.
CQRS/Vertical Slice
Các service feature-based: command/query/handler/controller theo từng use case.
Clean Architecture + DDD
Reward Ledger Service làm service lõi với Aggregate, Entity, Value Object, Domain Event.
Containerization
Docker Compose chạy toàn bộ service + database + Redis + RabbitMQ + MinIO.
2. Ý tưởng dự án EcoQuest Campus
EcoQuest Campus là hệ thống game hóa hành động xanh trong trường đại học. Sinh viên làm các hành động xanh như tái chế chai nhựa, check-in trạm xanh, tham gia cleanup event, báo cáo điểm có rác hoặc hoàn thành thử thách không dùng nhựa. Hệ thống xác minh hành động, cộng điểm, cấp huy hiệu, cập nhật bảng xếp hạng, phát chứng nhận và ghi nhận phần thưởng.

7.2. Luồng 1 - Xem nhiệm vụ xanh
1. Student mở EcoQuest Web.2. Web gọi API Gateway: GET /catalog/missions.3. Gateway route đến Green Catalog Service.4. Green Catalog đọc catalog_db và trả missions/stations/badges.5. Web hiển thị nhiệm vụ xanh.
7.3. Luồng 2 - Submit hành động xanh
1. Student chọn mission "Recycle Bottle".2. Student scan QR station A1 hoặc nhập station code demo.3. Web gửi draft đến Eco Action Service.4. Eco Action lưu draft/session vào Redis.5. Student bấm Submit.6. Eco Action kiểm tra idempotency key để tránh submit trùng.7. Eco Action gọi Verification Policy Service qua gRPC.8. Policy trả accepted=true, suggestedPoints=10.9. Eco Action lưu action vào MongoDB.10. Eco Action publish EcoActionAcceptedEvent lên RabbitMQ.
7.4. Luồng 3 - Cộng điểm, mở huy hiệu, cập nhật leaderboard
1. Reward Ledger consume EcoActionAcceptedEvent.2. Reward Ledger tạo RewardTransaction.3. RewardWallet aggregate cộng điểm.4. Reward Ledger kiểm tra điều kiện unlock badge.5. Reward Ledger lưu wallet/transaction/badge vào reward_db.6. Reward Ledger publish PointsGrantedEvent và BadgeUnlockedEvent.7. Leaderboard consume PointsGrantedEvent và update Redis Sorted Set.8. Frontend gọi Leaderboard Service để xem rank mới.
7.5. Luồng 4 - Kết thúc tuần/tháng và cấp chứng nhận
1. Admin hoặc scheduled job gọi POST /leaderboards/seasons/{id}/close.2. Leaderboard Service lấy top N từ Redis Sorted Set.3. Leaderboard lưu snapshot vào leaderboard_db.4. Leaderboard publish LeaderboardSeasonClosedEvent.5. Recognition Service consume event.6. Recognition Service tạo certificate PDF cho Top 3/Top 10.7. Recognition Service upload certificate vào MinIO.8. Recognition Service lưu metadata vào recognition_db.9. User vào Web để xem/download giấy chứng nhận.
7.6. Luồng 5 - Moderator duyệt action nghi ngờ
1. Verification Policy trả requiresManualReview=true.2. Eco Action lưu action với status=PENDING_REVIEW.3. Moderator mở màn hình review.4. Moderator approve hoặc reject.5. Nếu approve, Eco Action publish EcoActionAcceptedEvent.6. Nếu reject, Eco Action publish EcoActionRejectedEvent.7. Reward Ledger chỉ cộng điểm khi có accepted event.
